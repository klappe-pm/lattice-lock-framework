# Agent System Glossary & Reference

This document serves as the definitive reference for all variables, types, configuration options, and protocols used in the Vibelocity Orchestrator Agent System (v2.1).

## 1. Agent Configuration Variables

### Identity

- **`agent.identity.name`**: Unique identifier for the agent (kebab-case). Example: `model-orchestration-agent`.
- **`agent.identity.role`**: Human-readable job title. Example: "Chief Model Architect".
- **`agent.identity.version`**: Semantic versioning string. Example: "2.1.0".
- **`agent.identity.tags`**: List of keywords for discovery and categorization.

### Configuration

- **`agent.configuration.max_steps`**: Hard limit on the number of execution steps per task to prevent infinite loops. Default: `50`.
- **`agent.configuration.timeout_seconds`**: Maximum execution time before forced termination. Default: `300`.
- **`agent.configuration.log_level`**: Verbosity of logs. Options: `DEBUG`, `INFO`, `WARN`, `ERROR`.

## 2. Core Protocols

### Planning (`planning`)

- **`phases`**: The standard lifecycle of a complex task.
	- **`intent_analysis`**: Deciphering what the user actually wants.
	- **`information_gathering`**: collecting necessary context.
	- **`strategy_formulation`**: Deciding the high-level approach.
	- **`plan_generation`**: Creating the step-by-step execution plan.
	- **`validation`**: Checking the plan against constraints.
	- **`execution`**: Running the plan.
- **`default_strategy`**:
	- **`linear`**: Steps executed one after another.
	- **`parallel`**: Independent steps executed concurrently.
	- **`recursive`**: Task broken down into smaller identical sub-tasks.

### Estimation (`estimation`)

- **`complexity_weights`**: Multipliers for estimating effort.
	- **`code_modification`**: 1.5x base effort.
	- **`new_file_creation`**: 1.2x base effort.
	- **`read_only_analysis`**: 0.8x base effort.
- **`cost_limits`**:
	- **`warning_threshold`**: Cost level that triggers a user alert.
	- **`hard_limit`**: Cost level that stops execution immediately.

### Delegation (`delegation`)

- **`delegation_triggers`**: Conditions that cause a task to be handed off.
	- **`complexity_threshold`**: If task complexity score > X.
	- **`domain_mismatch`**: If task requires knowledge outside agent's domain.
	- **`resource_constraint`**: If agent is too busy or lacks context window.

## 3. Model Orchestration Types

### Task Types (`TaskType`)

Categorizes requests to optimize model selection.

- **`CODE_GENERATION`**: Writing new code. High precision required.
- **`DEBUGGING`**: Fixing errors. High reasoning required.
- **`ARCHITECTURAL_DESIGN`**: System planning. Large context required.
- **`DOCUMENTATION`**: Writing guides.
- **`TESTING`**: Creating tests.
- **`DATA_ANALYSIS`**: Interpreting data.
- **`REASONING`**: Complex logic.
- **`VISION`**: Image processing.

### Model Providers (`ModelProvider`)

- **`OPENAI`**: GPT-4o, O1, etc.
- **`ANTHROPIC`**: Claude 3.5 Sonnet, Opus.
- **`GOOGLE`**: Gemini 1.5/2.0 Pro, Flash.
- **`XAI`**: Grok models.
- **`OLLAMA`**: Local models (Llama 3, Qwen, DeepSeek).
- **`AZURE`**: Enterprise OpenAI.

### Selection Strategies

- **`quality_first`**: Maximize reasoning/coding score. Ignore cost.
- **`speed_priority`**: Maximize speed rating.
- **`cost_optimize`**: Minimize input/output cost. Prefer local models.
- **`balanced`**: Weighted average of all factors (Default).

## 4. Local Model Automation Variables

Variables for managing the local Ollama instance.

- **`RAM_BUFFER_MB`**: Minimum RAM to keep free for OS (Default: 1024 MB).
- **`KEEP_ALIVE_MODELS`**: List of models to keep loaded in VRAM.
- **`MAX_PARALLEL_MODELS`**: Maximum number of concurrent model instances.
- **`OLLAMA_HOST`**: URL of the Ollama API (Default: `http://localhost:11434`).

## 5. Agent Relationships

- **`Parent Agent`**: Coordinator. Responsible for planning, delegation, and final validation.
- **`Sub-Agent`**: Specialist. Responsible for executing specific scoped tasks.
- **`Sibling Agent`**: Agents at the same level that can communicate via the parent.

## 6. Output Formats

- **`JSON`**: Structured data for programmatic use.
- **`Markdown`**: Human-readable text with formatting.
- **`YAML`**: Configuration data.
- **`Artifact`**: A file generated by the agent (code, image, document).
